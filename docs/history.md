# record 開発の経緯 — ADR を読むための背景資料

作成: 2026-07-09（Claude による作業記録。各判断の詳細は [adr/](adr/) を参照）

このプロジェクトは 3 日間で設計判断が 3 回転した。ADR だけ読むと「なぜそんな回り道を
したのか」が見えないので、時系列の文脈をここに残す。

## 前史: ChatLog（〜2026-07-06）

前身は Electron アプリ **ChatLog**。アプリ内の xterm.js タブで claude を起動し、
U1（asciicastLogger = 生バイト記録）と U2（u2recon = frame-stitch 可読化）で
ログを取っていた。技術的には動いたが、「新しいウィンドウを開かず、いま使っている
端末の中で録りたい」という要件には合わず、CLI ツールとして作り直すことになった。
ChatLog リポジトリは参考資料として凍結（編集禁止）。

このとき既に分かっていたこと:
- claude の JSONL（`~/.claude/projects/`）が会話内容の最も忠実な記録である
- `asciinema convert -f txt` は claude の alt-screen TUI では会話が消失する（実測）
- 可読化は自作するしかない

## 第 1 期: scroll-commit エンジン（2026-07-07）

CLI 版 record を新規作成（ADR-0001, 0002）。実装言語はこのとき明示的に検討されず、
参照実装 ChatLog（JavaScript + xterm.js）の流れで Node.js になった — 後日ユーザーから
違和感の表明があり、事後評価して追認した（ADR-0007）。当初は ChatLog の frame-stitch を移植した
が、実 claude セッションでの検証で「同一行の畳み込み」（AAAA×26 が 1 行になる）と
「インデント・アンカー喪失」が判明し、その日のうちに **scroll-commit 方式**へ全面
置換した（ADR-0003）。「スクロールで破棄される瞬間の行を捕まえる」という発想で、
26 行中 25 行を復元。ここで見つけた実装の罠（xterm の onLineFeed はスクロール適用後で
手遅れ／claude は終了時に DECSTBM をリセットする）は ADR-0003 に記録した。

この時点の残課題:「長い実開発会話での品質確認」。

## 第 2 期: 実戦投入と綻び（2026-07-08〜09 午前）

実開発セッション（17 分、1783 行の表のペースト、履歴スクロール、ウィンドウリサイズ
まで含む「意地悪な」実データ）を変換したところ、冒頭に 42 行の順序の狂った重複断片が
出た。cast のイベントを時刻単位で解析して原因を特定 — 送信前の入力ボックス内
プレビューのスクロールを scroll-commit が本文として拾っていた。制御シーケンスでは
本文と区別できず、**背景色**（claude はユーザー入力系を #373737 で描く）と
**ブロック先頭のアンカー有無**の組み合わせで解決した（ADR-0004）。

続いてユーザーが 80×24 の小さい端末でテストし、より根本的な限界が露呈した。
小端末では claude がスクロールをほぼ使わず（SU がセッション全体で 9 回）、
しかも**速い応答の中間部分を端末に一度も描画しない**（40 項目中 19 項目が cast の
生ストリームに不存在）。これはどんな端末レコーダでも取れない。「scroll-commit の
改良」では越えられない壁だと確定した瞬間だった。

## 転機: 外部調査（2026-07-09 午前〜昼）

ユーザーが ChatGPT と並行検討した記録（[research/](research/) の 1.txt〜10.txt、
奇数=質問・偶数=回答）を検証した。ChatGPT の提案（script / tmux pipe-pane / capture-pane
差分 / control mode / pyte スナップショット）はいずれも「端末エミュレータ＋確定判定が
最終的に必要」に帰着し、record が既にその終着点にいることを確認した一方、
**`--ax-screen-reader` フラグの存在**という決定的な情報が得られた。

ax モードを実測すると期待通りだった: alt-screen 不使用、`you:`/`claude:` プレフィックス、
スピナー自己消去。ここで「**scrollback に押し出された行＝確定**」という劇的に単純な
確定ロジックを実装した（後の ADR-0005 の中核）。80×24 でも完全復元。

ところが**ユーザーが ax モードの操作感を却下**（「かなり使いにくい。alt-screen 導入前の
claude と同じではない」）。ax はスクリーンリーダー用の割り切った描画であり、
日常の対話には向かなかった。技術的に完璧でも UX で不採用 — この往復が
次の発見につながった。

## 第 3 期: classic レンダラーの発見と確定（2026-07-09 午後）

ax のドキュメントにあった「classic レンダラーを強制するため `tui` 設定は無効」という
一文から `tui` 設定を調査。公式ドキュメントは `true`/`false` と書いているが実際は
`"default"`/`"fullscreen"` で（false を渡すと SettingsError ダイアログ。そこにパイプ
入力すると「Fix with Claude」を誤選択する罠まで踏んだ）、**`"tui": "default"` が
まさにユーザーの求めていた「alt-screen 導入前の旧来型 TUI」**だった。見た目・操作感は
従来のまま、会話は scrollback に流れる。

ax 用に作った scrollback 到達コミットがそのまま効く。プロファイルを増やす代わりに
「alt-screen を一度も見なければ classic」という動的判定（sawAlt）を入れ、tui 設定が
どこで行われても対応できるようにした（ADR-0005）。実運用初回（80×34）で、
scroll-commit 時代の鬼門だった A×50 行が 50/50、選択肢クイズの Q&A も保存され、
ノイズ 0 を確認。

## エコシステムの整理（2026-07-09 夕）

record 単体の話から、ログ資産全体の役割分担に議論が移った（ADR-0006）。
ユーザーの手元には自作ツールが既に 2 つある:

- **vsc-claude-threadbox** — JSONL ビューア（ライブ更新・横断検索・Markdown 出力）
- **vsc-harness-logger** — VS Code 内で CLI を起動して端末ログを保存する拡張
  （README に自ら「alt-screen は最後の 1 画面しか残らない、raw キャプチャ層は
  別リポジトリで対応予定」と書いてあり、その"別リポジトリ"が record になった。
  なお classic モードの発見により、この拡張も `tui: "default"` にするだけで
  会話全部が残るようになる）

最終的な体制: **JSONL が正本**（Threadbox / CCHV で閲覧）、**record の .txt が
リアルタイム**（切実な必須要件）、**.cast は保険**（パターン崩れ時の再変換原本＋
JSONL に写らない一瞬の画面の記録）。asciinema バイナリへの依存はない。

ユーザーの根本方針: **作業ログは資産であり、技術負債の削減に役立てる**。
ツールは取り替え可能な閲覧レンズ、データ（プレーンなローカルファイル）が本体。
CCHV はメンテが止まったら fork してでも維持する、という覚悟も表明されている。

## 教訓（次の開発者=未来の自分たちへ）

1. **実データで殴られてから設計が決まる**。frame-stitch も scroll-commit も、綺麗な
   テストでは通り、実セッション（巨大ペースト、小端末、履歴スクロール）で壊れた。
   品質改善のたびに過去の .cast で回帰検証できたことが唯一の安全網だった。
2. **上流の描画モードを変える方が、下流の解釈を賢くするより強い**。scroll-commit の
   精緻化に注いだ努力より、`tui: "default"` の 1 行設定の方が品質に効いた。
   ただし前者があったから後者の価値を即座に測定できた。
3. **技術的に完璧でも UX で落ちる**（ax モード）。却下の過程で得た情報が次の発見の
   鍵になったので、回り道も無駄ではなかった。
4. **公式ドキュメントは実装に遅れる**（tui: true/false の乖離）。信じるのは実測。
