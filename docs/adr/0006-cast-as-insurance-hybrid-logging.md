# ADR-0006: ハイブリッドログ戦略 — JSONL を正本、.cast を保険とする役割分担

- ステータス: 採用
- 日付: 2026-07-09

## コンテキスト

ユーザーの目的は「過去の経緯の振り返り」と「まとめてコピーして再質問」。
そして基本方針として**作業ログは資産**（技術負債の削減に長期的に役立てる）。

ログ関連の道具が複数並立している:

- claude 自身の JSONL（`~/.claude/projects/`）— 構造化された会話の正本
- vsc-claude-threadbox（自作 VS Code 拡張）— JSONL のビューア（ライブ更新・検索・
  Markdown エクスポート）
- CCHV（claude-code-history-viewer、外部 OSS）— 25 種の AI CLI 履歴の横断ビューア
- vsc-harness-logger（自作 VS Code 拡張）— VS Code 内で CLI を起動して端末ログを保存
- record（本ツール）— 端末内記録（.cast + .txt）

「.cast は再生に実時間がかかり、コピーにも使えないので不要では」という問いが出た。

## 決定

各データ・ツールの役割を以下に固定する:

| 層 | 役割 |
|---|---|
| claude JSONL | **会話内容の正本**。振り返り・検索・コピーはここから（Threadbox / CCHV で閲覧） |
| record `.txt` | **リアルタイム閲覧**（切実な必須要件）。VS Code で開いて追記を追う。claude 以外の CLI の可読ログもこれ |
| record `.cast` | **保険**。能動的に開くものではない。(1) chrome パターンが崩れた時に修正後のエンジンで再変換する原本、(2) JSONL に写らないもの（一瞬の選択肢・権限プロンプト・画面表示そのもの）の唯一の記録 |
| asciinema バイナリ | **依存しない**（record は形式を自前で読み書きする）。`asciinema play` したい時だけ任意で使う |

.cast は「見る・コピーする」ためのものではない。可読テキストが欲しければ
`record convert`（一瞬で変換、再生時間ゼロ）を使う。

## 理由

- 資産の本体は**ツールではなくデータ**。JSONL・.cast・.txt はすべてプレーンな
  ローカルファイルなので、閲覧ツール（Threadbox / CCHV）が廃れても資産は残る。
  ツールは取り替え可能な「閲覧レンズ」に格下げしておく。
- .cast の保険価値は実証済み: 2026-07-07〜09 の品質改善はすべて過去の .cast の
  再変換で検証できた（.txt しか無ければ手遅れだった）。コストは 1 セッション
  数百 KB と記録時の tee のみで、ほぼゼロ。
- claude の UI はバージョンごとに変わる前提に立つ（ADR-0002）。パターン崩れは
  「いつか必ず起きる」ので、原本なしの運用は資産方針と矛盾する。

## 結果

- CCHV は「claude 以外の CLI の履歴も横断したくなったら」導入（メンテが止まったら
  fork して自力維持する方針をユーザーが表明。MIT なので可能）。
- ディスクが問題になったら `--no-cast` や自動削除ポリシーを検討する（現時点では
  プロファイルが枯れていないため全量保持）。
