# ADR-0007: 実装言語としての JavaScript (Node.js) — 事後追認

- ステータス: 採用（事後追認）
- 日付: 2026-07-09（実質の決定は 2026-07-07、明示的な検討なしに行われた）

## コンテキスト

**言語は発注時に指定されていなかった。** ユーザーが参考として挙げた前身プロジェクト
ChatLog（Electron + xterm.js）が JavaScript だったため、その流れで自然に
JavaScript (Node.js, ESM) になった — というのが正直な経緯であり、当時
「CLI のテキスト処理ツールに何の言語が最適か」という比較検討は行われていない。

ユーザーからは後日、率直な違和感が表明された:
「コマンドラインで動かすツールで、メインはテキスト処理なのに JavaScript を使うのは、
古い世代の感覚では違和感がある」。伝統的にこの種のツールは C / Perl / Python /
シェルの領分であり、その感覚は正当である。

本 ADR はこの暗黙の決定を明文化し、事後的に妥当性を評価して追認するものである。

## 決定

JavaScript (Node.js) を実装言語として **追認**する。書き直しは行わない。

## 事後評価 — 結果的に妥当だった理由

1. **決定的依存は端末エミュレータであり、xterm.js が最良だった。**
   可読化エンジンの品質は端末エミュレーションの正確さで決まる。xterm.js は
   VS Code が本番使用する最も実戦的な端末実装で、ヘッドレス版（`@xterm/headless`）と
   **パーサフックの公開 API（`registerCsiHandler`）** を持つ。scroll-commit 方式
   （ADR-0003）は「SU 適用**前**に割り込む」フックに依存しており、これができる
   エミュレータライブラリは他言語にほぼ無い（Python の pyte、Rust の avt/vt100、
   Go の vt10x はいずれも解釈結果を読む API が中心で、シーケンス適用前への割込みは
   想定外）。言語が xterm.js を選んだのではなく、**xterm.js が言語を選んだ**。
2. **前身の知見がそのまま移植できた。** ChatLog で実証済みの U1/U2 のロジックと、
   xterm.js の罠（onLineFeed の発火タイミング等）の知識が直接活きた。
3. **ツールチェーン全体が同一言語になった。** 記録対象の claude Code 自体が Node 製、
   ユーザーの自作ツール（vsc-harness-logger / vsc-claude-threadbox）も VS Code 拡張
   （TypeScript）。`reconstruct.mjs`＋`profiles.mjs` は**同一ランタイムなので
   VS Code 拡張にそのままライブラリとして移植できる**（構想済み）。
4. **性能は問題にならない。** 扱うストリームは 1 セッション数百 KB。テキスト処理の
   言語間性能差が意味を持つ規模ではない。
5. 時代的文脈: AI エージェント CLI の主流（claude Code、gemini-cli 等）が Node/TS で
   書かれており、「CLI ツール = JS」はこの領域では既に普通になっている。
   ユーザー自身の総括 —「時代といえば時代」。

## 代替案と、今日ゼロから選ぶ場合

- **Rust**: asciinema 3.x 自体が Rust で、同系の avt がある。単一バイナリ配布は魅力
  だが、パーサフック API の不在が scroll-commit の実装を困難にする。
- **Python**: pyte は簡易で、DECSTBM/SU の細部やフックが不足。
- **Go**: vt10x 等はあるがメンテが薄い。
- 結論: 今日ゼロから検討しても、fullscreen TUI の可読化を要件に含む限り
  xterm.js → Node.js に帰着する可能性が高い。classic モード（ADR-0005）だけなら
  他言語でも書けるが、フォールバック（ADR-0003）を持つ意味が消える。

## 結果・代償

- node-pty がネイティブ依存（記録モードのみ。convert 系は pure JS で全 OS 可搬）。
- 単一バイナリ配布はできない（Node ランタイムと node_modules が必要）。
- **教訓として**: 言語のような基盤選定は、参照実装からの惰性で決まりがちである。
  今回は結果的に技術的必然と一致していたが、本来は着手時に一度立ち止まって
  本 ADR のような比較を行い、発注者に選択肢を示すべきだった。
