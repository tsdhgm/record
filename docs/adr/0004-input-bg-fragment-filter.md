# ADR-0004: 入力背景色フィルタによるペースト断片の除去

- ステータス: 採用（fullscreen モードの scroll-commit 経路に適用）
- 日付: 2026-07-09

## コンテキスト

実開発セッション（17 分、1783 行の表をペースト）の品質確認で、変換結果の冒頭に
**42 行の順序の狂った重複断片**が見つかった。cast のイベント解析で原因を特定:
巨大テキストをペーストすると、**送信前の入力ボックス内プレビュー**がスクロールし、
scroll-commit フック（ADR-0003）がその断片を本文として拾ってしまう。送信後には
claude が正規のエコー（`❯` 付き全文）を改めて流すため、純粋な重複になる。

制御シーケンスの構造だけでは判別できないことも確認した（プレビューのスクロールと
本文のスクロールは同一のシーケンスパターン）。

## 決定

claude がユーザー入力系を**専用の背景色**で描く事実（実測: 入力ボックス／送信済み
エコー = `#373737`、bash モード `!` = `#413c41`、ダークテーマ）を判別に使う:

> 入力背景色を持つ行の連続ブロックは、**先頭行がアンカー（`❯` / `!`）で始まる場合
> のみ本文（送信済みエコー）として残し、アンカー無しで始まるブロックは入力ボックス
> 内プレビューの断片として捨てる。**

コミット時に xterm のバッファセルから背景色を読み（非空白セルの過半で判定）、
ブロック状態は O(1) の状態機械 1 個（`inputRun: null|keep|drop`、空行では状態維持）
で追跡する。リアルタイム動作を損なわない。

## 理由

- 背景色単独では不可: 送信後の正規エコーも同じ `#373737` で描かれる。
- アンカー単独では不可: エコーの継続行（2 行目以降）には ❯ が付かない。
- 「ブロック先頭のアンカー有無」は原理に根ざす: 画面上部から順に破棄される以上、
  正規エコーの破棄は必ず ❯ 行から始まる。プレビュー断片は途中からしか現れない。

## 結果

- 実測: 断片 42 行を全除去、正規エコーは無傷、既存テスト（AAAA×26）に回帰なし。
- 副次効果として、履歴スクロール中の残骸のうち入力背景のものも抑制される。
- 制約: 背景色はテーマ依存（ライトテーマ値は未計測）。値は `profiles.mjs` の
  `inputBgs` にあり、テーマ違いで断片が混入したら追記する。
- 「Jump to bottom」インジケータ付きの行は履歴スクロール中の再描画だが、
  **一次コミットである場合がある**ため行ごと捨ててはいけない（実測で穴が開くことを
  確認済み）。サフィックスの除去のみ行う。
