# ADR-0002: 可読化エンジンの自作とプロファイル層の分離

- ステータス: 採用
- 日付: 2026-07-07

## コンテキスト

ログの主目的は「人間があとで読む・検索する・コピーする」こと。`.cast` 原本は
制御コードだらけで直接読めないため、可読テキストへの変換が必要。しかも
**リアルタイム**（記録中に追記される）が必須要件。

既存の変換手段を実測評価した:

- `asciinema convert -f txt`（3.x）: 最終画面バッファのダンプ。claude のような
  alt-screen TUI では **会話がほぼ全て消失**（冒頭ダイアログしか残らない）。
- 制御コードの正規表現除去（`perl -pe 's/\e...//'` 等）: カーソル移動・上書き・
  再描画が解釈されないため、断片と重複だらけで読めない。
- 端末エミュレータで画面復元＋定期スナップショット（pyte 等の一般的な提案）:
  「現在画面」しか出せず、追記型ログにならない。スナップショットを溜めると
  同一内容が大量に重複する。

## 決定

1. `@xterm/headless`（本物の xterm.js のヘッドレス版）に生バイトを食わせて画面状態を
   正確に復元し、**「行が確定した瞬間」を検出してその行だけを追記する**エンジンを自作する。
   確定検出の方式は ADR-0003（scroll-commit）→ ADR-0005（scrollback 到達）で発展。
2. ツール固有の知識（UI ノイズ = chrome のパターン、アンカー文字、画面レイアウト、
   背景色）はエンジンから分離し、**プロファイル層**（`lib/profiles.mjs`）に閉じ込める。
   エンジンはツール非依存を保つ。

## 理由

- 「読みやすい追記ログ」には画面状態の解釈が不可欠で、既存ツールにその層が存在しない
  （2026-07-09 の外部調査でも同結論。ChatGPT の提案も最終的に「端末エミュレータ＋
  確定判定の自作が必要」に帰着した）。
- xterm.js は claude 自身が前提にしている端末実装に最も近く、DECSTBM・SU・alt-screen
  等の解釈が信頼できる。パーサフックの公開 API（registerCsiHandler）がある。
- claude の UI はバージョンごとに変わる。壊れるのはパターンであってエンジンではない、
  という構造にしておけば、修正はプロファイル 1 ファイルで済み、`.cast` 原本から
  再変換すれば過去分も救える。

## 結果

- claude / claude-ax / generic の 3 プロファイルで運用（2026-07-09 時点）。
- chrome パターンは claude 更新で崩れ得ることを既知の制約として README に明記。
- 品質向上の作業はほぼ全てプロファイルと確定判定の改良に集約され、記録層は不変。
